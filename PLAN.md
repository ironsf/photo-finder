# photoFinder — план

Программа под Windows (Python) для сбора картинок товаров по штрихкоду с ручной
проверкой «да/нет».

## Что делает

1. Читает `.xlsx` (штрихкод в столбце A, названия — в остальных, часть на румынском).
2. По каждому товару ищет картинки (по штрихкоду и/или названию, см. стратегию поиска ниже).
3. Показывает человеку картинку + название → «Да» сохраняет, «Нет» показывает следующий вариант.
4. Приводит выбранную картинку к разумному размеру (см. «Обработка изображения» ниже, пропорции
   сохраняются, фон не трогаем — кроме заливки белым при прозрачности) → `images/<code>.jpg`.

## Пример реального файла (проверено)

Формат подтверждён на присланном примере (`XLSX (2).xlsx`, 20 строк):

```
A1: 8059037078096   B1: Lampa de noapte 3D Cat iTotal
A2: 8053634001297   B2: Calculator Candy Macaron alb iTotal
...
```

- **Без строки заголовка** — данные с первой строки.
- Столбец A — EAN-13 (13 цифр, как текст — важно не терять ведущие нули).
- Столбец B — название на румынском, часто с брендом в конце (здесь везде **iTotal**).
- **Категория — подарки/сувениры/игрушки** (ночники, брелоки, пеналы, настольные игры, тапочки), НЕ еда/косметика.
  Это меняет расчёт по Tier-0 (см. ниже) — план обновлён под это.
- Один штрихкод (`2200002471848`, префикс `22`) — «restricted circulation» GS1-код (внутримагазинный/весовой),
  таких глобальные базы штрихкодов не знают в принципе — сразу уходит на поиск по названию.
- Найдено веб-поиском: товары реально продаются на румынских маркетплейсах (kidune.ro, nichiduta.ro,
  emag.ro, bellfyd.ro и др.) — это подсказывает эффективную стратегию поиска (см. ниже).

## Ключевые решения (согласованы)

| Вопрос | Решение |
|---|---|
| Обработка изображения | Диапазон 650–1050 (обе стороны) — вне диапазона ресайз до ~700, пропорции сохраняются |
| Объём | Тесты 10 → 100–1000 → прод ~14000 (не более) |
| Источники (водопад) | **Tier 0** бесплатные базы по EAN → **Tier 1** Brave → **Tier 2** SerpAPI |
| Перевод | Переводить название на **английский** для поиска |
| Интерфейс проверки | Окно Tkinter (встроено в Python) |

## Снижение стоимости: водопад источников (cheapest-first)

Главный принцип — **платный поиск последним, а не первым**. Каждый товар проходит источники по порядку
и останавливается на первом успешном:

```
Tier 0  Open Food/Beauty/Products Facts + UPCitemdb  → $0, фото по точному EAN (низкий охват для non-food)
Tier 1  Brave: name+brand → штрихкод                  → ~$0.005/запрос
Tier 2  SerpAPI / Google (фолбэк на хвост)            → платно, только остаток
```

⚠️ **Пересмотрено по реальному файлу:** каталог — подарки/сувениры (бренд iTotal), не еда/косметика.
Open Food/Beauty Facts заточены под FMCG и тут почти не сработают — Tier-0 остаётся (проверка бесплатна,
почему бы не попробовать), но **не закладываем в него экономию**, ожидаем близкое к 0% покрытие для этой
категории. UPCitemdb/Open Products Facts возможно дадут единичные попадания, тоже не считаем основной экономией.

- **Tier 0 (бесплатно, но охват ~0–10% для этого каталога):** официальные фото по штрихкоду.
  Картинки привязаны к точному EAN → почти всегда верные → **авто-приём без ручного «да/нет»** (опция).
  Open Food Facts / UPCitemdb — без ключа или free 100/день. Стоит $0 — оставляем как первый шаг всегда.
- **Tier 1 (Brave, основной источник для non-food):** $5/1000 (metered, $5/мес free-кредита ≈ 1000 запросов).
  **Порядок запроса меняем:** для этого каталога штрихкод как текст в поиске картинок работает плохо
  (сувенирка редко индексируется по цифрам EAN) — **запрос по названию+бренду эффективнее**, штрихкод —
  как второй/резервный вариант. См. стратегию поиска ниже.
- **Tier 2 (SerpAPI):** планами $25/1 000 · $75/5 000 · $150/15 000; вызывается **только** когда Brave пуст.

**Прочие бесплатные меры:** дедуп штрихкодов в файле; кэш навсегда (не искать дважды); эскалация на второй
запрос только при промахе; **сужение по известным ритейлерам** (см. ниже) — бесплатно повышает точность.

**Новый бесплатный рычаг — сужение по доменам ритейлеров.** Веб-поиском подтверждено: товары бренда
продаются на конкретных маркетплейсах (kidune.ro, nichiduta.ro, emag.ro, bellfyd.ro, giftday.ro и т.п.).
Если у Brave/SerpAPI задать `site:emag.ro OR site:kidune.ro ...` (или отсортировать результаты, отдавая
приоритет этим доменам), резко падает доля мусорных картинок — **бьёт прямо по жалобе «половина ошибочная»**,
и это ничего не стоит сверх обычного запроса. Список доменов пополняется по мере находок в первых прогонах.

**Оценка на 14 000 (пересмотрено под non-food каталог):**

| | Без доп. мер | С Tier-0 + сужением по доменам |
|---|---|---|
| Закрыто бесплатно (Tier-0) | 0% | ~0–10% |
| Итог | ~$70–95 | **~$65–90** (экономия скромнее, чем для FMCG) |
| Точность 1-го кандидата | базовая | заметно выше за счёт доменного сужения |
| Ручных проверок | ~14 000 | ~14 000 (Tier-0 почти не разгружает non-food) |

**Главный вывод:** для этого каталога основная экономия — не в Tier-0, а в **точности запроса**
(название+бренд+сужение по доменам), которая снижает число «нет» на товар и, как следствие, число
обращений к Tier-2. Бесплатный замер (прогон Tier-0 по всем 14 000, $0) всё равно стоит сделать —
он покажет реальный процент, а не оценку.

### ✅ Реальный тест на 20 товарах (Ф1 пройдена)

- **Tier-0:** 0/20 (0%) — подтверждено фактом, не оценкой. Для этого каталога бесполезен, оставлен
  в коде как бесплатная первая попытка на случай FMCG-позиций среди остальных 14 000.
- **Tier-1 (Brave, запрос = название с брендом):** **20/20 (100%)** получили кандидатов, `SerpAPI
  фолбэк не понадобился ни разу`. Топ-кандидаты в основном с реальных ритейлеров (emag.ro, nichiduta.ro),
  разрешение часто 720×720.
- **Визуальная проверка:** простой товар (ночник-кот) — точное совпадение с первой попытки; сложный
  товар (подарочная коробка с редким служебным штрихкодом) — картинка похожая, но общая/неспецифичная →
  именно такие случаи и должна ловить ручная проверка «да/нет», это не баг, а ожидаемое поведение.
- **Tier-2 (SerpAPI):** формат ответа подтверждён, интеграция протестирована 1 живым запросом (100
  кандидатов, парсинг корректный). Не понадобился на этой выборке, но готов на случай хвоста.
- **Вывод по деньгам:** на этом каталоге реальный расход близок к нижней границе оценки (~$70 на
  14 000, в основном Brave, почти без SerpAPI) — заметно лучше худшего сценария.

- **Перевод:** `deep-translator` (бесплатный бэкенд Google) = $0; официальный Translate API ≈ $10–12 разово.
- **Кэш и возобновление обязательны** — перезапуск не должен жечь квоту заново.
- **14000 ручных проверок — большой труд.** Авто-приём Tier-0 + горячие клавиши + предзагрузка; готовые пропускаются.
- Источники вынесены в абстракцию — Tier 2 при желании заменяется на Google Custom Search.

## Поток обработки одного товара

```
для каждого (code, name):
    если уже done/skipped в report.csv или есть images/<code>.jpg → пропустить (возобновление)
    кандидаты = search_candidates(code, name)   # Brave → SerpAPI, ранжирование по домену + SigLIP2-score
    показать кандидатов по одному в окне:
        если топ-кандидат: score ≥ 0.9 И не low-res → запустить авто-приём (обратный отсчёт N сек)
        "Да" (вручную или по таймеру) → обработать → сохранить → status=done → следующий товар
        "Нет"        → следующий кандидат (отменяет таймер)
        "Пропустить" → status=skipped → следующий товар
        "Назад"      → предыдущий кандидат
    если кандидаты кончились → status=notfound
    если платный бюджет исчерпан (MAX_PAID_SPEND_USD) → остановка с понятным сообщением
```

## ✅ Стратегия поиска (обновлено по ручному тестированию, `_build_queries` в `search.py`)

**Найденная проблема (реальный тест пользователя):** поиск только по названию для товаров-животных
(«Breloc Giraffe iTotal») давал в основном **сток-фото самого животного** (istockphoto, gettyimages,
dreamstime) — не товар, а жираф вообще. Штрихкод один эту проблему не решал (специфичные сувениры
плохо индексируются по цифрам).

**Решение — комбинированный запрос:** теперь пробуем `<название> <штрихкод>` вместе как основной запрос
(штрихкод в тексте действует как «якорь», отсекающий сток-фото и общий контент — Brave начинает возвращать
почти исключительно карточки реальных интернет-магазинов). Название и штрихкод по отдельности остаются как
запасные варианты, если комбинированный ничего не нашёл:

1. `<название> <штрихкод>` — основной запрос.
2. `<название>` — если комбинированный пуст.
3. `<штрихкод>` — последний вариант.

### ✅ Ленивая догрузка кандидатов (по итогам 2-го теста пользователя)

**Проблема:** Brave отдаёт максимум ~20 картинок за запрос (параметр `count` для image-поиска не
наращивает выдачу). Пользователь, отклонив несколько вариантов, упирался в конец списка и товар
помечался «не найдено».

**Решение — `build_search_plan` + ленивая подгрузка в UI:** строится план из пар (провайдер, запрос),
дешёвые первыми: **Brave** по всем 3 вариантам запроса → **SerpAPI/Google** по тем же → (опц.) **Яндекс**.
Первый шаг показывается сразу; каждый следующий шаг подгружается **только когда пользователь исчерпал
текущих кандидатов** (жмёт «нет» на последнем). Новые кандидаты дедуплицируются по URL и дописываются
в конец списка. Так пул растёт по мере «нет», а платятся только реально нужные вызовы.
**Проверено:** пул рос 20 → 39 → 89 → … → 382 уникальных кандидата (Brave 3 запроса + Google 3 запроса),
дубликаты корректно отсеиваются; в UI после отклонения всей первой порции автоматически подгрузилась вторая.

### ✅ Только крупные картинки, без апскейла (по итогам 2-го теста)

**Проблема:** программа находила мелкие картинки и растягивала (апскейлила) их до ~700 — мыло.

**Решение:**
- `image_proc.prepare_image` больше **никогда не увеличивает** картинку: только уменьшает те, что крупнее
  `MAX_DIMENSION` (1050) до `RESIZE_TARGET` (700); всё остальное сохраняется как есть.
- Кандидаты с известным размером мельче `MIN_IMAGE_DIMENSION` (650 по длинной стороне) помечаются
  `too_small`, уходят **в конец списка** и **не участвуют в авто-приёме**. Крупные показываются первыми.
- В UI в строке статуса виден размер (`WxH px`) и предупреждение «⚠ малое разрешение (крупнее не нашлось)»
  для мелких. Если у товара крупных фото нет вовсе — в консоль пишется `[no-large] <код>`.
- Порог `MIN_IMAGE_DIMENSION` настраивается в конфиге (для этого каталога много мелких стоков — большинство
  сувенирных фото в вебе 220–600px, поэтому крупных реально мало; это ограничение источников, не программы).

### Источники и Яндекс

Каждый уровень запроса пробуется сперва в Brave, затем в SerpAPI/Google. **Яндекс.Картинки** доступны
через SerpAPI (движок `yandex_images`, тот же ключ, платно) — включается флагом `USE_YANDEX_FALLBACK`
(по умолчанию выключен), срабатывает последним, когда Brave и Google ничего не дали. У самого Яндекса
пригодного бесплатного публичного API поиска картинок нет — только через SerpAPI-обёртку.

**Проверено:** для лёгких товаров (точное совпадение легко находится) результат не ухудшился — нужная
картинка по-прежнему первая. Для сложных (редкие сувениры-животные) комбинированный запрос убирает
сток-фото полностью, хотя не всегда находит именно нужный вариант из серии (например, из линейки
брелоков-животных может подтянуться не «жираф», а «кот»/«свинка» — сама серия слабо индексирована
по конкретному животному). **Замечено:** название в файле иногда на английском («Giraffe»), а сайты
продавца используют румынский («Girafa») — это тоже может мешать точному совпадению; полноценно не
решено (требует полноценного словаря англ.↔рум. соответствий животных/названий, не просто перевода).
Для таких «плохо гуглящихся» товаров SigLIP2 корректно даёт низкий score — авто-приём не сработает,
решение остаётся за человеком, что и является ожидаемым поведением системы.

## ✅ Обработка изображения (Ф2 пройдена, `image_proc.py`)

Отказались от жёсткого квадрата 650×650 (crop/letterbox) — лишняя работа и риск обрезать/испортить
кадр там, где это не нужно. Вместо этого — диапазон допустимых размеров:

- Скачать кандидата (с кэшем сырых байт в `cache/images/`, по хэшу URL — повторный клик «назад»/перезапуск
  не платит и не качает заново).
- **Только уменьшаем, никогда не увеличиваем** (обновлено по 2-му тесту, см. раздел стратегии поиска):
  - длинная сторона > `MAX_DIMENSION` (1050) → уменьшить до `RESIZE_TARGET` (700);
  - в диапазоне 650–1050 → оставить как есть;
  - мельче `MIN_IMAGE_DIMENSION` (650) → **оставить в исходном (мелком) размере, НЕ апскейлить**,
    пометить `too_small=True`.
- **Прозрачный фон → белый:** если у источника есть альфа-канал (PNG с прозрачностью), перед конвертацией
  в JPEG накладываем на белую подложку — закрывает пожелание «нейтральный белый фон» без ИИ-вырезания.
- EXIF-ориентация исправляется автоматически (`ImageOps.exif_transpose`).
- Сохранение — JPEG качества 90 → `images/<code>.jpg`.

**Проверено:** мелкие 200×200 / 500×500 → остаются как есть (без апскейла), помечены `too_small`; 800×800 →
без изменений; 1600×1200 → уменьшено до 700×525; PNG с прозрачностью → плоский белый фон, размер сохранён.

## ✅ Интерфейс Tkinter (Ф3 пройдена, `review_ui.py`)

- Показывает: превью картинки (уменьшенный `thumbnail_url`, полноразмерная скачивается только при «Да»),
  штрихкод + название, счётчик «Товар N/total · кандидат k/M», домен источника.
- **Штрихкод и название — в полях (readonly Entry): выделяются мышью и копируются (Ctrl+C).** Клик по
  штрихкоду копирует его в буфер целиком и показывает всплывающее «✓ Скопировано в буфер обмена»
  (гаснет через 1.5с). Проверено headless: клик → буфер, уведомление появляется и само исчезает.
- Кнопки и горячие клавиши: **Да** [Y/Enter] → обработка через `image_proc` → сохранение → следующий товар;
  **Нет** [N/→] → следующий кандидат; **Пропустить** [S] → следующий товар без сохранения;
  **Назад** [B/←] → предыдущий кандидат; **Открыть в браузере** → страница-источник; **Esc/закрытие окна** → выход.
- Предзагрузка следующего кандидата в фоновом потоке (`threading`), с диск-кэшем скачанных превью.
- Уже готовые товары (есть `images/<code>.jpg`) пропускаются автоматически при следующем запуске.
- **Живой прогон подтверждён пользователем:** Пропустить + 3× Да отработали корректно на реальных данных
  (реальные файлы сохранены через весь пайплайн поиск→обработка→сохранение), окно закрывается штатно.
- Требует `python-tk` (на Windows Tkinter идёт в комплекте официального установщика Python; на macOS для
  Homebrew Python нужно доп. `brew install python-tk@<версия>`).
- Перевод названия в UI пока не показывается (отдельная строка появится в Ф4).

## ⏸ Playwright/Bing-скрейпинг (испытано, приостановлено)

Идея: бесплатный поиск через Playwright + браузер вместо платных API, раз человек всё равно проверяет
каждую картинку. Реализовано (`playwright_search.py`, Chromium через Playwright), но при живом тестировании
оказалось **нестабильным**: одна и та же выдача Bing то отдавала 60–70 кандидатов, то всего 1 нерелевантный
результат без видимой закономерности (похоже на A/B-варианты страницы или анти-бот throttling). Для
необслуживаемого прогона на 14 000 это неприемлемый риск. По ходу нашли и исправили два реальных бага
(non-encoded query URL, обрезка кандидатов до ранжирования по домену), но саму нестабильность подгрузки
результатов не победили за разумное время.

**Решение:** модуль оставлен в проекте (рабочий, кэш/структура готовы), но выключен по умолчанию
(`USE_SCRAPE_SEARCH = False`). Можно вернуться к отладке позже; сейчас поиск идёт через Brave/SerpAPI,
которые уже доказали 100% покрытие на тестовой выборке.

## ✅ SigLIP2: скоринг релевантности + авто-приём (`relevance.py`)

Вместо экономии на поиске (не удалось через Playwright) — экономим **время ручной проверки** и повышаем
точность первого кандидата, используя локальную AI-модель (бесплатно, работает на CPU, без API).

- **Модель:** `google/siglip2-base-patch16-224` через `transformers` (torch, CPU). Мультиязычная —
  работает с румынским текстом напрямую, перевод (Ф4) не обязателен для скоринга.
- **Как считается:** для топ-N кандидатов (`RELEVANCE_TOP_N = 10`) параллельно скачиваются превью
  (ThreadPoolExecutor), затем **один батч-проход** модели даёт sigmoid-вероятность соответствия
  картинка↔название для каждого сразу (эффективнее, чем по одному).
- **Ранжирование:** кандидаты пересортировываются по score (высокий — первым), домен-эвристика
  (`rank_candidates`) применяется до этого как первичная грубая сортировка.
- **Авто-приём с обратным отсчётом:** если топ-кандидат имеет `score ≥ AUTO_ACCEPT_THRESHOLD` (0.9)
  **и** разрешение источника не низкое (`не low-res`, защита от чёткого-по-смыслу, но смазанного апскейла) —
  запускается таймер (`AUTO_ACCEPT_DELAY_SECONDS`, по умолчанию 4с) с видимым обратным отсчётом в UI.
  Любое действие человека (Да/Нет/Назад/Пропустить/Открыть в браузере) отменяет таймер. Если человек
  ничего не нажал — картинка сохраняется автоматически, `auto_accepted=True` в отчёте для аудита.
- **Проверено на реальных данных:** точное совпадение (ночник-кот) → 75–99% в разных тестах; заведомо
  неверная картинка (подарочная коробка вместо ночника) → 0.00006%; неоднозначные товары (брелоки-животные
  редких форм) → закономерно низкий score (1–3%), корректно уходят на ручную проверку без авто-приёма.
  Low-res guard подтверждён: кандидат 150×150 со score 99.6% **не** был авто-принят.
- **Задержка:** первый запуск модели ~5с (одноразово при старте программы). Дальше на товар — обычно
  2–5с (в основном скачивание превью, не сама модель) — синхронно, до показа первого кандидата. Если на
  практике это будет раздражать — можно вынести в фон по аналогии с предзагрузкой превью (не делали, чтобы
  не усложнять раньше времени).
- **Зависимости:** `torch` + `transformers` (CPU-версии) + вес модели ~400–800 МБ, скачивается один раз
  с Hugging Face при первом запуске.

## Конфиг (config.py) — фактическое состояние

- `INPUT_XLSX` — путь; по умолчанию первый `*.xlsx` в папке скрипта.
- `BARCODE_COL = "A"`, `NAME_COL = "B"`, `HAS_HEADER = False` — подтверждено на примере файла.
- `OUTPUT_DIR = images/`, `CACHE_DIR = cache/`.
- `MIN_DIMENSION = 650`, `MAX_DIMENSION = 1050`, `RESIZE_TARGET = 700`, `JPEG_QUALITY = 90`.
- `MIN_IMAGE_DIMENSION = 650` — порог «крупной» картинки (длинная сторона); мельче → в конец, без апскейла.
- `BRAVE_API_KEY`, `SERPAPI_KEY` — читаются из `.env` (не в коде, файл в `.gitignore`).
- `CANDIDATES_PER_QUERY = 50` (Brave фактически отдаёт ~20; больше добирается ленивой догрузкой),
  `RETAILER_DOMAINS = [...]` — приоритет доменов при ранжировании.
- `REPORT_CSV`, `SPEND_FILE`, `BRAVE_COST_PER_CALL = 0.005`, `SERPAPI_COST_PER_CALL = 0.025`,
  `MAX_PAID_SPEND_USD = 20.0` — потолок суммарного платного расхода (растёт вместе с прогоном, не сбрасывается).
- `SEARCH_PROVIDERS = ["google", "brave"]` — **порядок и включение источников** (список = приоритет
  сверху вниз; убрать из списка = выключить). Доступно: `google` (SerpAPI, дороже/качественнее),
  `brave` (дешевле), `yandex` (SerpAPI), `bing` (Playwright, нестабильно). По умолчанию Google первым
  (Brave показал себя хуже в ручном тесте). `SCRAPE_HEADLESS = True` — только для `bing`.
- `QUERY_ORDER = ["barcode", "name_barcode", "name"]` — **что и в каком порядке ищем**. По умолчанию
  сперва только штрихкод (как оператор ищет вручную), затем «название штрихкод», затем название.
  Ещё доступно `"name_en"` (перевод названия на англ.).
- `TRANSLATE_TO_ENGLISH = False` — перевод названия на английский для запроса `name_en`
  (нужен пакет `deep-translator`).
- `USE_RELEVANCE_SCORING = True`, `RELEVANCE_MODEL_ID`, `RELEVANCE_TOP_N = 12`,
  `AUTO_ACCEPT_THRESHOLD = 0.9`, `AUTO_ACCEPT_DELAY_SECONDS = 4`.

⚠️ **Стоимость при Google-первым:** первый запрос на товар теперь идёт в SerpAPI (~$0.025), а не в Brave
(~$0.005). На ~14 000 товаров это ≈ $350 против ≈ $70. Если нужно дешевле — поставьте `"brave"` первым
в `SEARCH_PROVIDERS`, Google оставьте вторым (фолбэком).

**Сброс прогресса:** `python reset.py` (удаляет `report.csv` и `images/*.jpg`, кэш и бюджет не трогает;
`--all` — очистить и кэш поиска; `--yes` — без подтверждения).

## ✅ Возобновление, отчёт и бюджет (Ф5 пройдена, `state.py`)

- `report.csv`: `code, name, status(done|skipped|notfound|error), image_url, domain, relevance,
  auto_accepted, width, height, low_res, error_msg, timestamp`. Пишется построчно (append), без перезаписи
  всего файла — безопасно при аварийном закрытии.
- При старте пропускаются товары со `status ∈ {done, skipped}` в отчёте **или** с уже существующим
  `images/<code>.jpg` (двойная защита). `notfound`/`error` — не считаются финальными, при повторном
  запуске попытка повторяется автоматически.
- **Guard по платному бюджету:** каждый реальный (не кэшированный) вызов Brave/SerpAPI прибавляет оценочную
  стоимость в `cache/spend.json`; при достижении `MAX_PAID_SPEND_USD` — `QuotaExceededError` с понятным
  сообщением, интерфейс останавливается штатно (не крашится), сумму можно поднять в конфиге и продолжить.
- **Проверено:** многократные тесты возобновления (повторный запуск верно пропускает уже решённые товары
  и продолжает с нужного места), срабатывание guard по бюджету, запись/чтение всех полей отчёта.

## Этапы реализации

- ✅ **Ф0 — Каркас.** `config.py` + `excel_reader.py` + `main.py`. Проверено на 20 реальных строках.
- ✅ **Ф1 — Источники.** `barcode_db.py` (Tier-0, 0/20) → `search.py` (Tier-1 Brave 20/20 + Tier-2

- ✅ **Ф2 — Картинки.** `image_proc.py`: скачивание+кэш, условный ресайз 650–1050→~700, флаг LOW-RES,
  заливка белым при прозрачности, сохранение JPG. Проверено на 20 реальных кандидатах.
- ✅ **Ф3 — UI.** `review_ui.py`: окно Tkinter Да/Нет/Пропустить/Назад + горячие клавиши + счётчик +
  предзагрузка. `main.py` теперь запускает интерфейс. Подтверждено живым прогоном пользователем.
- ⏸ **Ф4 — Перевод.** Отложена по решению пользователя. SigLIP2 мультиязычна и работает с румынским
  напрямую, так что перевод сейчас не блокирует качество скоринга — актуальность ниже, чем казалось раньше.
- ✅ **Ф5 — Состояние.** `state.py`: report.csv, возобновление, guard по платному бюджету. Дополнительно

## Открытые вопросы / риски

- **Столбцы названий:** ✅ подтверждено на примере — A=штрихкод, B=название, без заголовка.
- **Смесь категорий:** пример файла — весь один бренд/категория (сувениры). Если в полном каталоге 14000
  будут вперемешку FMCG и non-food, стоит уточнить долю каждого — Tier-0 может дать эффект на FMCG-части.
- **Ключи API:** ✅ есть — `BRAVE_API_KEY` и `SERPAPI_KEY` в `.env`.
- **Список доменов ритейлеров:** стоит пополнить `RETAILER_DOMAINS` по факту прогона на большем объёме —
  это бесплатно поднимает точность ранжирования (уже частично закрыто SigLIP2-скорингом, но домены
  остаются полезным дополнительным сигналом).
- **Задержка ~2–5с на товар** от relevance-скоринга — приемлемо на 20 товарах, стоит перепроверить
  ощущение на 100–1000 перед боевым прогоном на 14 000; если раздражает — вынести в фон (Ф6).
- **Torch на Windows:** нужно явно ставить CPU-версию (см. requirements выше), иначе рискуем скачать
  многогигабайтный CUDA-билд без пользы (CPU и так работает).
- **Юр. нюанс:** картинки из поиска защищены авторским правом; для внутреннего каталога обычно приемлемо, но стоит держать в уме.
- **Объём ручной проверки:** авто-приём при высокой уверенности снижает нагрузку на уверенных товарах;
  на неоднозначных (низкий/средний score) — по-прежнему нужен человек на каждый.
